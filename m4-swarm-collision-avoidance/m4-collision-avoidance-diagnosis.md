## 1. 避碰演算法家族地圖

先看全局：這些方法大致分兩軸——**反應式 vs. 預測式**、**幾何解析 vs. 優化求解**。

```
反應式（只看當下）───────────────────────► 預測式（看未來一段時間）
APF（純幾何力場）    VO/RVO/ORCA（幾何+速度空間）    DMPC / CBF-QP（優化求解）
    │                        │                              │
 最便宜、最不穩          業界最成熟、有理論保證          最精準但最貴（求解時間）
```

| 方法 | 核心概念 | 計算成本 | 已知失效模式 |
|---|---|---|---|
| **APF**（Khatib, 1986） | 目標=吸引力、障礙/鄰機=排斥力，合力方向即為速度指令 | 極低（O(n)，純代數運算） | **局部極小值**（合力抵銷、卡死不動）；兩機正面接近時的**震盪/來回擺盪**（reciprocal dance problem）；增益（gain）調參高度敏感，換場景就要重調 |
| **VO / RVO / ORCA**（van den Berg et al., ICRA 2008 / ISRR 2011） | 在「速度空間」算出會導致碰撞的速度集合，選一個安全速度；ORCA 讓兩機**各自負責一半**閃避責任，化簡成低維度線性規劃 | 低（O(n) 個線性規劃，逐 pair 計算） | 假設**互惠性（reciprocity）**——所有鄰機都同樣遵循 ORCA、且都誠實廣播真實速度；異質機隊（有的機種遵循、有的不遵循）或感測誤差大時保證會失效；密集場景仍可能**卡死（deadlock）** |
| **BVC（緩衝 Voronoi 胞）**（Zhou, Wang, Bandyopadhyay, Schwager, RA-L 2017） | 每台機用「與鄰機安全距離的一半」內縮出自己的 Voronoi 胞，只要軌跡留在自己胞內，**只需感測相對位置、不需通訊速度/意圖**即可保證不相撞 | 低（O(n)，類似 ORCA 量級） | 內縮換取的安全裕度**偏保守**，密集隊形容易卡死/凍結；理論保證建立在單積分器（速度直接可控）+ 純位置感測，複雜動力學需額外擴充 |
| **DMPC（分散式模型預測控制）** | 每台機求解一個有限時域最優控制問題，把「不相撞」寫成約束、把「跟隨隊形/任務」寫成代價函數，逐步往前滾動求解 | 高（每個控制週期都要跑優化，20 機即時求解對機載算力是挑戰） | 通訊延遲/丟包會讓各機基於「過期的鄰機預測」求解，導致實際軌跡違反約束；優化問題在密集場景可能**無可行解（infeasible）**，需要 fallback 策略 |
| **CBF-QP（控制屏障函數）**（Ames et al., ECC 2019） | 把「安全集合」定義成一個屏障函數，即時求解一個小型 QP，在**最小修改**原始控制指令的前提下保證系統軌跡不離開安全集合 | 中（比 DMPC 便宜，QP 比完整優化問題快很多，可即時跑） | 屏障函數/相對階數設計不當會導致解不存在或過度保守；多機情境需要每個 pair 一條約束，數量隨機數平方成長；常與 MPC/RL 混合以兼顧安全與任務表現 |
| **學習式 / RL（CADRL 系列）**（Chen, Liu, Everett, How, ICRA 2017；Everett, Chen, How, IROS 2018） | 用強化學習訓練一個價值網路，把「自己狀態 + 可觀測到的鄰機狀態」映射到偏好速度，去中心化、不需通訊 | 訓練貴、**推論便宜**（一次前向傳播） | **沒有形式化的無碰撞保證**，只有訓練分佈內的統計安全；機數遠超訓練時、或幾何佈局在分佈外時性能會掉；有 sim-to-real gap |

**白話比喻**
- APF 像「用兩塊磁鐵互相排斥導航」——簡單但容易在角落卡住或抖動。
- ORCA 像「開車時默契禮讓」——雙方各退一步，前提是**對方也遵守交通規則**。
- BVC 像「每人有自己的停車格，畫線畫保守一點」——安全但格子太密會擠不進去。
- DMPC 像「每一步都重新規劃一小段未來的路線」——精準但要花時間思考，思考太久車就已經開過頭了（延遲問題）。
- CBF-QP 像「方向盤裝了一個安全氣囊式的即時修正器」——你想怎麼開隨你，但它會用最小力道把你拉回安全範圍內。
- CADRL 像「靠經驗直覺開車的老司機」——平常很順，但遇到沒見過的路況（訓練沒看過的密集度/佈局）可能失準。

## 2. 為什麼「群飛避碰在真實世界裡常常不穩定/不精準」——工程面共同病灶

理論上每個演算法都有漂亮的收斂/安全證明，但那些證明幾乎都假設「完美狀態、零延遲、無雜訊」。真實系統會在下面幾個地方漏氣：

| 病灶 | 說明 | 對哪些演算法影響最大 |
|---|---|---|
| **狀態估計/定位雜訊傳播** | GPS 多路徑、視覺里程計漂移、EKF 融合延遲，讓每台機「自己以為的位置」跟「真實位置」有落差；避碰演算法吃到帶雜訊的輸入，安全邊界形同虛設 | ORCA/BVC（理論建立在精確相對位置上）> APF > DMPC/CBF（可加穩健餘裕但更保守） |
| **通訊延遲/丟包** | 20 台機廣播位置/速度，網路壅塞或封包遺失讓每台機拿到的是「過期」的鄰機狀態 | ORCA（互惠假設要求對方的速度資訊即時）、DMPC（規劃基於鄰機預測軌跡，過期預測=規劃無效） |
| **離散化誤差** | 連續時間的安全保證離散成固定控制週期（例如 20Hz）後，兩機有可能在**取樣間隔之間**就已經違反安全距離，尤其高速接近時 | 所有方法都有，速度越快、控制週期越長，風險越大 |
| **對稱佈局的死鎖/震盪** | 兩機正面對衝、或多機在瓶頸處同時到達，演算法給出的「安全解」在對稱條件下可能無唯一解，出現原地震盪或卡死 | APF（局部極小值）、ORCA/BVC（密集場景已知會 deadlock）尤其明顯 |
| **參數調校對場景敏感** | APF 的排斥力增益、ORCA 的時間視界（time horizon）、CBF 的類別 K 函數斜率——這些參數在模擬里調得漂亮，換一個實際場景（風、機型、機隊密度）就不準 | APF 最敏感，DMPC/CBF 次之，ORCA/BVC 相對穩健（參數少） |
| **時間同步** | 20 台機的內部時鐘/控制迴圈若沒對齊，"同時刻"的狀態廣播其實帶有隱含的時間差，等同放大了通訊延遲的問題 | 所有分散式方法都受影響 |
| **機隊異質性** | 如果不是所有機都跑同一套避碰邏輯（例如 leader 用不同的軌跡生成方式），ORCA 之類依賴「互惠假設」的方法保證會直接失效 | ORCA/RVO 系列首當其衝 |

**讀到這裡停一下 ⏸**：把這張表對照你現有的 ROS2 模組自問——「不穩定」比較像是**演算法本身在密集/對稱場景下的固有缺陷**，還是**上面這幾個工程病灶**造成的？多數團隊第一次做避碰時，90% 的「不穩定」其實是**通訊延遲/時間同步/QoS 設定**這種工程問題，而不是演算法選錯——這也是為什麼下一份 [ROS2 工具鏈筆記](m4-ros2-swarm-tooling.md) 值得優先排查。

## 3. 症狀 ↔ 診斷對照表（拿你現有模組直接比對）

| 你觀察到的症狀 | 最可能原因 | 排查方向 |
|---|---|---|
| 兩機接近時**來回擺盪/抖動**，不會撞但很難看 | APF 局部極小值/reciprocal dance；或 ORCA 時間視界設太短，速度指令變化過於劇烈 | 檢查是否用純 APF；若用 ORCA，拉長 time horizon 並加上速度平滑/限制加速度 |
| 兩機在瓶頸/走廊處**同時卡住不動** | 對稱佈局死鎖，ORCA/BVC/APF 都有已知案例 | 加入非對稱擾動（微小隨機偏移打破對稱）或優先權排序（誰先動由 ID/任務優先度決定） |
| 密集隊形時避碰**過度保守、繞遠路** | BVC 內縮過多、或安全距離設太大 | 檢查安全距離參數是否用了「最壞情況」而非「機型實際尺寸+合理誤差」 |
| 高速接近時**偶發擦撞或距離小於安全值** | 離散化誤差、控制週期太慢、或狀態估計延遲 | 提高避碰迴圈頻率；檢查位置/速度的 timestamp 與實際延遲，補償預測 |
| 部分機況良好、部分機**反應遲鈍/軌跡跳動** | 通訊丟包導致該機拿到的鄰機狀態過期；或該機的定位品質本身較差 | 檢查 DDS QoS 設定與該機的丟包率、定位來源（見 M4③） |
| 模擬穩定、實機不穩定 | 典型 sim-to-real gap：模擬沒有的定位雜訊、通訊延遲、風擾動 | 在模擬中人為注入雜訊/延遲，重新驗證演算法魯棒性再上機 |
| 20 台一起飛比 5 台一起飛時明顯變差 | 演算法複雜度隨機數增加（DMPC 尤其明顯），或 DDS 廣播量增加造成的通訊壅塞 | 檢查每個控制週期的實際求解/計算耗時是否隨機數線性外的成長；檢查網路頻寬是否飽和 |

## 4. 給你的實務建議

以你目前「已有 ROS2 模組但不穩定精準」的狀況，優先順序建議：

1. **先做工程面體檢，別急著換演算法**：對照 §2/§3，把通訊延遲、QoS、時間同步這幾項排除，很多時候「演算法沒問題，是餵給它的資料有問題」。
2. **ORCA/RVO2 是目前最值得作為基線（baseline）的方法**：開源實作成熟（RVO2 Library）、有 15 年以上的實戰驗證（含多篇擴展到四旋翼/非全向機器人的論文，見 [M4④ 論文清單](m4-papers-reading-list.md)），計算成本低到可以在機載算力即時跑。如果你現有模組不是 ORCA 系列，值得先拿 ORCA 當作對照組，量化比較「不穩定」是不是換了演算法就消失。
3. **BVC 是 ORCA 的一個實用替代/補充**：如果你的機隊感測不到彼此速度、只能感測相對位置（例如純視覺/UWB 測距），BVC 不需要通訊意圖就能保證安全，值得評估。
4. **CBF-QP 適合當「安全氣囊層」疊加在你既有的隊形/任務軌跡之上**：不用整個換掉現有邏輯，而是在最外層加一個「最小修正」的安全過濾器——這通常是從「堪用但不穩定」進化到「穩定精準」性價比最高的一步。
5. **DMPC 先別急著上**：對 20 台機即時求解的計算負擔大，除非你已經有餘裕的機載算力與良好的通訊基礎，否則優先度排最後。
6. **務必先在模擬中注入雜訊/延遲再驗證**：PX4 SITL 多機模擬 + 人為加入定位雜訊與通訊延遲，是低成本找出「哪些不穩定是演算法固有、哪些是工程問題」的辦法（見 [M4③](m4-ros2-swarm-tooling.md)）。
